# Plan 3 — Search CLI (Rust hot-path)

> **Status:** SHIPPED 2026-04-19

**Goal:** Implement the real `odin-search` query path. End state: pointing the binary at a pre-built `content/index/` returns ranked JSON or human-readable results in <50 ms cold (warm Ollama) on the GPU desktop. All CLI flags from spec §5.5 work. Auto-detect logic falls back to keyword-only when Ollama is unreachable or `enable_vector_search=false`.

**Depends on:** Plan 1 (Rust crate + clap surface exist; `tools/search/src/main.rs` has stub branches to fill in). Plan 2 is **not** strictly required to land first — Plan 3 can be developed against hand-built fixture indices checked into `tools/search/tests/fixtures/`.

**Architecture:** Single static binary, no daemon. Per spec §5.1: `mmap` USearch + `open` SQLite, embed via Ollama HTTP, run BM25 (FTS5) + HNSW (USearch) in parallel via rayon, fuse via Reciprocal Rank Fusion, fetch top-N rows from SQLite, serialize. Each layer is a separate module so swaps are local (USearch → sqlite-vec, Ollama → fastembed-rs, etc.).

**Tech Stack:** Rust stable; clap (already from Plan 1); `rusqlite` with `bundled` + `feature = "fts5"` features; `usearch` crate (Rust bindings to the C++ index); `reqwest` with `blocking` feature (Ollama HTTP); `serde` + `serde_json`; `rayon` (two queries in parallel); `toml` (settings.toml parsing); `dirs` (resolve `~/.config/odin_master/`).

## Components (rough)

1. **Settings loader** — read `~/.config/odin_master/settings.toml`; defaults match spec §2.5; missing file is fine (treated as defaults).
2. **Index opener** — accept index dir path (default `./content/index/`), open `chunks.sqlite`, mmap `vectors.usearch`. Fail clearly if neither exists; degrade to FTS5-only if just USearch is missing.
3. **Embed client** — blocking HTTP call to Ollama; sends the spec §5.3 instruction prefix (`"Instruct: Given a web search query, retrieve relevant passages...\nQuery:{query}"`); times out at a configurable threshold (default 5 s); on failure, returns `Result::Err` so `main` can fall back.
4. **FTS5 query** — escape user query for FTS5 MATCH syntax; apply tier/source/view filters via SQL WHERE; return ranked `(row_id, bm25_score)` list.
5. **HNSW query** — call USearch search with the query vector; apply label-based filters; return `(chunk_id, vector_score)` list.
6. **Fusion** — RRF combine the two ranked lists; return final top-N.
7. **Row fetch + JSON output** — single SQLite query for top-N rows; format per spec §5.5 schema.
8. **Auto-detect at startup** — check vector index presence, settings flag, Ollama reachability; pick mode; print one-line stderr note when falling back. Lazy-start Ollama (per spec §5.5) is best-effort; failure is silent and we continue keyword-only.
9. **Doctor** — `odin-search --doctor` reports: settings file path + parsed values, index dir path + sizes, Ollama URL + reachability, pulled models (via `/api/tags`), warm-vs-cold latency probe.

## Test gates (acceptance criteria)

- Snapshot tests on JSON output for 5–10 fixed queries against fixture indices: byte-for-byte stable across runs.
- Latency benchmark in the test suite: warm <50 ms, cold <100 ms on dev box (the cold target accounts for Ollama embed call). Bench is an `#[ignore]`d test runnable via `cargo test --release -- --ignored bench`.
- Each filter flag (`--tier`, `--source`, `--view`, `--top`) demonstrably narrows results in a snapshot test.
- `--keyword-only` and `--fast` skip the embed call entirely (verified by pointing at a mock-Ollama server and asserting zero hits).
- `--semantic-only` skips the FTS5 path.
- Ollama unreachable → falls back to keyword-only with a one-line stderr warning; exits 0.
- Missing index dir → exits non-zero with a clear message naming the expected path.
- `--doctor` output is stable + machine-grep-able.
- FTS5 query escape handles malicious input (e.g., `query"; DROP TABLE chunks --`) without breaking.
